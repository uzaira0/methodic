# Low-Priority Security Findings — Recommendations

## L-3: Sensitive IDs in URL Query Parameters
**File:** `chronicle-web/src/containers/survey/SurveyContainer.js`
**Status:** Accepted Risk

Participant-facing pages (survey, time-use-diary) use `?studyId=<uuid>&participantId=<string>` in URLs.
These are visible in browser history and server logs.

**Mitigating factors:**
- Study IDs are UUIDs (not sequential/guessable)
- Participant IDs are researcher-assigned strings
- These pages are intentionally unauthenticated (participant-facing)
- No sensitive data is returned by the API — only study settings and survey questions

**Future recommendation:** Replace URL query params with short-lived, single-use tokens
generated by the backend. Participant receives a link like `/chronicle/survey?token=<opaque>`.
The backend resolves the token to studyId+participantId. This requires backend changes to
create and manage tokens with expiration.

## L-5: Outdated Frontend Dependencies
**File:** `chronicle-web/package.json`
**Status:** Requires coordinated update with testing

Outdated packages with known vulnerabilities:
- `axios ~0.26.0` — multiple CVEs (SSRF, prototype pollution). Update to `~1.7.0+`
- `auth0-lock ~11.33.3` — EOL. Consider migrating to `@auth0/auth0-spa-js` or `@auth0/auth0-react`
- `react ~16.14.0` — EOL. Update to React 18+ (requires testing of all components)
- `eslint ~7.32.0` — outdated, update to 8.x+
- `jsonwebtoken ~8.5.0` (devDep) — known algorithm confusion CVE. Update to 9.x+

**Impact of blind update:** HIGH — React 16→18 migration requires testing all components,
removing enzyme (use React Testing Library), and updating react-router to v6.
Auth0 Lock is deeply integrated into the auth flow.

**Recommended approach:**
1. Update `axios` first (minimal breaking changes): `~1.7.0`
2. Update `jsonwebtoken` devDep: `~9.0.0`
3. Plan React 18 migration as a separate sprint
4. Plan auth0-lock → @auth0/auth0-spa-js migration as a separate sprint

## L-7: TDE File-Based Key Provider on Same Host
**Status:** Infrastructure recommendation

The pg_tde keyring directory (`/var/lib/postgresql/tde-keyring`) is on the same host
as the encrypted data. If an attacker gains access to the host filesystem, they can
read both the data files and the encryption keys.

**Current mitigation:** Docker named volume isolation, root ownership on keyring directory.

**Recommended improvements (in order of preference):**
1. **HashiCorp Vault integration**: Use pg_tde's Vault key provider instead of file-based.
   The Vault instance is already deployed in the compose stack.
   ```
   SELECT pg_tde_add_key_provider_vault_v2(
     'chronicle-vault-provider',
     'http://vault:8200',
     '<vault-token>',
     'secret/data/chronicle/tde'
   );
   ```
2. **Separate mount**: Store keyring on a different volume/filesystem with restricted access.
3. **Key rotation**: Implement periodic key rotation using pg_tde's key rotation functions.
