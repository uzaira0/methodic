# Chronicle Cryptographic Audit — April 2026

**Audit Date:** 2026-04-03
**Auditor:** Automated cryptographic audit (Claude)
**Scope:** All Chronicle submodules — server, Android client, web frontend, rhizome framework, Docker/deployment infrastructure
**Context:** Google's April 2026 announcement of a more feasible Shor's algorithm implementation with fewer qubits accelerates the timeline for RSA/ECC deprecation.

---

## Executive Summary

Chronicle's cryptographic posture is **generally strong for classical threats** but has **significant quantum-vulnerability exposure** in its TLS infrastructure and certificate management. The application's core authentication (JWT HS256/HMAC-SHA256) is quantum-safe. The primary risks are:

1. **RSA-based TLS certificates** used throughout (Nginx, PostgreSQL SSL, Traefik) — broken by Shor's algorithm
2. **ECDHE key exchange** in TLS cipher suites — broken by Shor's algorithm
3. **Self-signed SSL cert generation script uses RSA-2048** — undersized even for classical threats
4. **Backup encryption uses AES-256-CBC** — functional but AES-GCM preferred for authenticated encryption
5. **PKCS12 keystore uses hardcoded password "changeit"** — credential hygiene issue

No broken classical algorithms (MD5, SHA-1, DES, RC4, Blowfish) were found in production code.

## 2. Quantum Vulnerability Assessment

### 2.1 Algorithms Broken by Shor's Algorithm

Shor's algorithm (and its 2026 optimized variants) can efficiently factor large integers and compute discrete logarithms, breaking:

| Algorithm | Where Used | Impact | Severity |
|-----------|-----------|--------|----------|
| **RSA** (all key sizes) | TLS certificates (Nginx, PostgreSQL, Traefik), init-ssl.sh (RSA-2048), init-postgres-ssl.sh (RSA-4096) | Attacker can derive private keys from public certificates, enabling MITM on all TLS connections | **CRITICAL** |
| **ECDHE** (all curves) | All TLS cipher suites (ECDHE-RSA-*, ECDHE-ECDSA-*) | Attacker can break key exchange, compromising forward secrecy for all TLS sessions | **CRITICAL** |
| **ECDSA** (secp384r1) | Defined in init-postgres-ssl.sh as alternative | Not currently in use, but would be broken if adopted | **INFO** |
| **DHE-RSA** | Nginx prod config cipher suite | Diffie-Hellman key exchange vulnerable to Shor's | **CRITICAL** |

### 2.2 Algorithms Resistant to Quantum Attacks

| Algorithm | Where Used | Quantum Impact | Status |
|-----------|-----------|---------------|--------|
| **AES-256-GCM** | Android KeyStore, EncryptedSharedPreferences | Grover's reduces to ~128-bit equivalent — still secure | **SAFE** |
| **AES-256-CBC** | Backup encryption, SQLCipher | Same as above, though GCM preferred | **SAFE** |
| **HMAC-SHA256** | JWT signing (HS256), mobile API signatures | Symmetric — not affected by Shor's | **SAFE** |
| **SHA-256** | Body hashing, backup checksums, SCRAM-SHA-256 | Grover's impact marginal for collision resistance | **SAFE** |
| **PBKDF2** | Backup key derivation | Symmetric KDF — quantum-safe | **SAFE** |
| **SCRAM-SHA-256** | PostgreSQL authentication | Symmetric challenge-response — quantum-safe | **SAFE** |

### 2.3 "Harvest Now, Decrypt Later" Risk

The primary risk is not immediate decryption but **retroactive decryption**:
- An adversary recording TLS-encrypted traffic today can store it and decrypt it once a cryptographically relevant quantum computer (CRQC) is available.
- Chronicle handles **research participant data (PII, health-related)** under HIPAA — this data has long-term sensitivity.
- All ECDHE-based TLS sessions are at risk even though they provide forward secrecy against classical adversaries.

## 3. Detailed Findings and Recommended Migrations

### CRITICAL Findings

#### C-1: RSA-2048 Self-Signed Certificate (init-ssl.sh:35)

```
openssl req -x509 -nodes -days 365 -newkey rsa:2048
```

**Issue:** RSA-2048 is below NIST's current minimum recommendation of RSA-3072 for classical security, and is completely broken by Shor's.

**Migration:**
- **Immediate:** Increase to RSA-4096 minimum for classical security (consistent with init-postgres-ssl.sh which already uses 4096).
- **Planned (2026-2027):** Migrate to hybrid certificates (classical + PQC) once Let's Encrypt and browser support mature.

#### C-2: All TLS Certificates Use RSA

All PostgreSQL SSL certs (CA, server, client) and the Nginx/Traefik production certs use RSA key pairs signed with SHA256WithRSAEncryption.

**Migration:**
- **Immediate:** No action needed for classical security (RSA-4096 is adequate).
- **Planned (2027):** Transition to hybrid PQC certificates as tooling matures.

#### C-3: TLS Cipher Suites Rely Exclusively on ECDHE/DHE Key Exchange

All configured cipher suites (nginx.prod.conf, traefik-tls.yml, postgresql-ssl.conf) use ECDHE or DHE key exchange, both vulnerable to Shor's.

**Migration:**
- **Planned (2027):** Adopt hybrid PQC-classical key exchange once support is available:
  - Nginx: Awaiting OpenSSL 3.5+ with PQC provider (expected Q3 2026)
  - Traefik: Awaiting Go 1.24+ PQC cipher suites (experimental in Go 1.23)
  - PostgreSQL: Awaiting libpq PQC support

### WARN Findings

#### W-1: Backup Encryption Uses AES-256-CBC (backup-chronicle.sh:76)

```bash
openssl enc -aes-256-cbc -salt -pbkdf2 -iter 100000
```

**Issue:** AES-CBC provides confidentiality but not authenticated encryption. An attacker could modify encrypted backups without detection (bit-flipping attacks).

**Recommendation:** Migrate to AES-256-GCM:
```bash
openssl enc -aes-256-gcm -salt -pbkdf2 -iter 100000
```
**Note:** OpenSSL `enc` command had limited GCM support historically. Verify with OpenSSL 3.x+. Alternatively, use `age` or `gpg --aead` for authenticated encryption.

**Priority:** Planned

#### W-2: PKCS12 Keystore Hardcoded Password (init-postgres-ssl.sh:262)

```bash
-password pass:changeit
```

**Issue:** The PKCS12 client keystore is generated with the well-known default password "changeit". If this keystore is deployed to production, it provides minimal protection.

**Recommendation:** Generate a random password and store it securely (Vault, Docker secrets).

**Priority:** Planned

#### W-3: PBKDF2 with 100k Iterations (backup-chronicle.sh:76)

**Issue:** While 100,000 PBKDF2 iterations is acceptable, NIST SP 800-132 (2023 update) recommends a minimum of 600,000 iterations for PBKDF2-SHA256. Argon2id would be stronger for key derivation.

**Recommendation:** Increase to 600,000 iterations or switch to Argon2id if the backup toolchain supports it.

**Priority:** Watch

#### W-4: TLS 1.2 Minimum (Not TLS 1.3-Only)

All TLS configurations set `minVersion: TLSv1.2`. TLS 1.3 is preferred as it:
- Removes RSA key transport (only ECDHE/DHE)
- Simplifies cipher suite negotiation
- Will be the first protocol version to get PQC hybrid key exchange

**Recommendation:** Set minimum to TLS 1.3 where client compatibility allows. At minimum, ensure TLS 1.3 is enabled and preferred (it already is in all configurations).

**Priority:** Planned

#### W-5: Nginx Development Config Has No TLS (nginx.conf)

The development `nginx.conf` listens on port 80 only, with no TLS. This is acceptable for local development but should never be used in production.

**Status:** Acceptable (dev-only), production uses `nginx.prod.conf` with TLS.

## 4. Post-Quantum Cryptography Roadmap

### 4.1 NIST PQC Standards (Finalized 2024)

| Standard | Algorithm | Use Case | Chronicle Relevance |
|----------|-----------|----------|-------------------|
| **ML-KEM** (FIPS 203) | CRYSTALS-Kyber | Key encapsulation (replaces ECDHE/DH) | TLS key exchange |
| **ML-DSA** (FIPS 204) | CRYSTALS-Dilithium | Digital signatures (replaces RSA/ECDSA) | TLS certificates, code signing |
| **SLH-DSA** (FIPS 205) | SPHINCS+ | Hash-based signatures (conservative fallback) | Certificate backup option |

### 4.2 Library Availability (as of April 2026)

| Library | Language | PQC Support | Status |
|---------|----------|-------------|--------|
| **OpenSSL 3.5** | C | ML-KEM, ML-DSA via oqs-provider | Stable, production-ready |
| **BouncyCastle 1.79+** | Java/Kotlin | ML-KEM, ML-DSA, SLH-DSA | Stable, usable for server-side |
| **Go crypto (1.24+)** | Go | ML-KEM experimental | Experimental, Traefik dependency |
| **AWS-LC** | C | ML-KEM hybrid | Production-ready (FIPS validated) |
| **liboqs** (Open Quantum Safe) | C | Full NIST suite | Reference implementation, usable |
| **Android Conscrypt** | Java/Android | Not yet | No PQC support as of Android 16 |

## 7. What Chronicle Gets Right

The audit should also note strong practices already in place:

1. **JWT uses HMAC-SHA256 (HS256)** — symmetric signing is quantum-safe. No RSA or ECDSA JWT signatures.
2. **AES-256-GCM for Android data encryption** — gold standard, quantum-safe (128-bit equivalent under Grover's).
3. **httpOnly Secure SameSite=Strict cookies** — proper JWT transport, not stored in localStorage.
4. **HMAC request signing with replay prevention** — mobile API uses HMAC-SHA256 with nonce and timestamp validation.
5. **SCRAM-SHA-256 for PostgreSQL authentication** — quantum-safe, constant-time password verification.
6. **TDE (Transparent Data Encryption)** for data at rest — with proper key management via HashiCorp Vault option.
7. **Strong security headers** — CSP, HSTS, X-Frame-Options, Permissions-Policy all properly configured.
8. **CSRF protection** — double-submit pattern with SameSite cookies.
9. **4096-bit RSA keys for PostgreSQL SSL** — strongest classical RSA, buys time for PQC migration.
10. **Encrypted backup pipeline** — AES-256 encryption with PBKDF2 key derivation, SHA-256 integrity checks.
